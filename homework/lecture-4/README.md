# RAG Pipeline — Вопрос-ответная система по книге

Retrieval-Augmented Generation пайплайн для ответов на вопросы по русскоязычным книгам. Ключевая особенность v5 — автоматическая Map-Reduce суммаризация глав через LLM вместо захардкоженных саммари.

## Архитектура

```
Книга (.txt)
  │
  ▼
StructureParser          — парсинг на части и главы
  │
  ▼
BookSummarizer           — Map-Reduce суммаризация (с кэшем)
  │                        ├─ Map: суммаризация каждой главы (2-3 предложения)
  │                        ├─ Reduce: общее резюме книги (5-7 предложений)
  │                        └─ Извлечение алиасов персонажей
  ▼
ChunkProcessor           — sentence-aware разбиение на чанки (spaCy)
  │
  ▼
ChromaDB + BM25 Index    — индексация (dense embeddings + лексический индекс)
  │
  ▼
┌─────────── Запрос пользователя ───────────┐
│                                           │
│  QueryExpander → расширение запроса        │
│  алиасами персонажей                      │
│                                           │
│  HybridRetriever:                         │
│    1. Semantic search (ChromaDB)          │
│    2. BM25 (лексический поиск)            │
│    3. Cross-encoder reranking             │
│       + metadata boost (NER)              │
│                                           │
│  GigaChat → генерация ответа              │
└───────────────────────────────────────────┘
```

## Компоненты

| Класс | Назначение |
|---|---|
| **StructureParser** | Парсит текст книги в структуру `{part, chapter, text}`. Поддерживает римские (`I`, `II`) и арабские (`1. Название`) номера глав |
| **BookSummarizer** | Map-Reduce суммаризация через GigaChat. Кэширует результат в `book_summaries.json` (инвалидация по SHA-256 хешу файла) |
| **MetadataExtractor** | Извлекает персонажей (PER) и локации (LOC) через spaCy NER с поддержкой алиасов |
| **QueryExpander** | Расширяет запрос алиасами персонажей для улучшения BM25-поиска |
| **ChunkProcessor** | Разбивает текст на чанки с учётом границ предложений и глав, с перекрытием |
| **HybridRetriever** | Гибридный поиск (semantic + BM25) → дедупликация → cross-encoder reranking с метаданными |
| **RAGPipeline** | Оркестратор: загрузка книги, декомпозиция запроса, поиск, генерация ответа |

## Используемые модели и технологии

- **GigaChat** — LLM для суммаризации, декомпозиции запросов и генерации ответов
- **BAAI/bge-m3** — dense embeddings (FP16)
- **BAAI/bge-reranker-v2-m3** — cross-encoder reranker (FP16)
- **ChromaDB** — векторная БД (in-memory, cosine distance)
- **BM25Okapi** — лексический поиск
- **spaCy** (`ru_core_news_sm`) — сегментация предложений, NER

## Установка

```bash
pip install -r requirements.txt
python -m spacy download ru_core_news_sm
```

Создайте файл `.env` в директории проекта:

```
GIGACHAT_CREDENTIALS=<ваш_токен>
```

## Запуск

```bash
python rag_pipeline_v5.py
```

Пайплайн загрузит книгу `book.txt`, построит индекс и запустит интерактивный цикл:

```
Ваш вопрос: Как звали главного героя книги?
Ответ: ...

Ваш вопрос: выход
```

Команды выхода: `выход`, `exit`, `quit`.

## Ключевые параметры

| Параметр | Значение | Описание |
|---|---|---|
| `chunk_size` | 900 | Размер чанка в символах |
| `sentence_overlap` | 2 | Перекрытие в предложениях между чанками |
| `top_k` | 12 | Количество финальных результатов |
| `candidate_pool_size` | 15 | Кандидатов из каждого метода поиска |
| `metadata_boost` | 0.1 | Вес метаданных (NER) при reranking |

## Как работает поиск

1. **Декомпозиция запроса** — LLM решает, нужно ли разбить сложный вопрос на подзапросы
2. **Query Expansion** — если в запросе упомянут персонаж, добавляются его алиасы
3. **Hybrid Retrieval** — параллельный поиск по семантике (ChromaDB) и ключевым словам (BM25)
4. **Дедупликация** — объединение кандидатов из обоих источников
5. **Reranking** — cross-encoder оценивает пары (запрос, чанк), метаданные (совпадение персонажей/локаций) дают бонус к score
6. **Генерация** — GigaChat формирует ответ по найденным фрагментам с инструкцией не выдумывать

## Ограничения

### Структура книги
- Парсер поддерживает только два формата глав: римские цифры (`I`, `II`, ..., `X`) и арабские с названием (`1. Название главы`). Другие форматы (`Глава 1`, `Chapter 1:`, `1)`) не распознаются
- Римские цифры ограничены I–X — книги с 11+ главами в одной части потеряют главы XI и далее
- Части должны называться `Часть первая/вторая/.../шестая` — книги без явного деления на части или с другим форматом (`Part 1`, `Часть I`) не распарсятся
- Контент до первой распознанной главы (предисловие, пролог, эпиграф) игнорируется

### Язык
- Парсер структуры работает только с русскоязычными заголовками
- NER-модель (`ru_core_news_sm`) заточена под русский язык — на других языках качество извлечения персонажей и локаций резко падает
- Промпты для GigaChat написаны на русском

### Масштаб
- ChromaDB используется in-memory — при очень больших книгах может не хватить оперативной памяти
- Суммаризация вызывает LLM на каждую главу — для книг с большим количеством глав это занимает значительное время и расходует токены

## Кэширование

Суммаризация кэшируется в `book_summaries.json`. При изменении файла книги (по SHA-256) кэш автоматически обновляется.
